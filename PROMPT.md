## 1. 프로젝트 설계 시
```
kotlin, spring boot, gradle을 사용하는 환경에서 사이드 프로젝트를 진행할 거야

프로젝트의 기본 기능은 아래와 같아
1. 사용자의 회원 가입 API
2. 관리자가 사용할 API : 가입한 회원의 조회, 수정, 삭제 기능
3. 사용자의 로그인 API : 자신의 상세 정보 조회 기능

기능에 대한 조건은 관리자 API의 인증 수단은 basic auth 기반이어야 한다는 거야

프로젝트 구조, 기능에 대한 고민
1. 내 생각엔 subproject로 나눌 필요는 없을 것 같아
2. 고민이 되는 지점은 domain과 orm(jpa)로 package를 구분해서 domain의 data class로 비즈니스 로직을 만들지, 기능이 복잡하지 않기 때문에 단순하게 service 레이어에서 jpa entity를 사용하는 로직을 만들지야
3. 로그인 시에는 JWT를 사용해서 만료시간을 주는 게 어떨까 싶어
4. 보안, 필터에 spring security를 사용하는 것과 임의로 filter를 만들어서 사용하는 것 중 나은 방향
5. 사용자 정보, 관리자 정보 등은 간단하게 h2를 붙이는 게 어떨까?

내 프로젝트의 기본 기능에 대한 고민의 방향이 적절한지, 더 나은 방향이 있는지 설명해줘
만약 더 나은 방향일지 아닐지에 대한 정보가 부족하다면 다시 질문해도 좋아
```

## 3. 사용자 회원가입 API
```
지금은 사용자의 회원가입 API가 필요해 조건은 아래와 같아 

1. 사용자 입력 값은 계정/암호/성명/주민등록번호/핸드폰번호/주소 
2. 핸드폰 번호 11자리, 주민등록번호는 13자리 같은 길이는 고려해야 하지만, 실제 유효한 핸드폰 번호인지, 주민등록번호인지 등은 확인할 필요 없어 
3. 단, DB에서 사용자의 계정과 주민등록번호는 유일해야 해 
내 생각엔 주민등록번호는 생년월일만 사용하는 경우가 있어서 앞 6자리와 뒤 7자리가 구분되어 저장되어야 할 것 같아. 
뒷자리는 필요한 경우가 있을 수 있기 때문에 복호화를 고려한 암호화를 해야 할 것 같아. 
암호는 다시 확인할 필요가 없기 때문에 해시로 저장해도 될 것 같아 

주소는 도, 혹은 시, 군, 구 ... 처럼 큰 단위에서 작은 단위로 입력한다고 가정하고, '사용자가 로그인 해서 본인의 정보를 확인할 때, 가장 큰 단위의 정보를 보여줘야 해. 
그렇다면 가장 큰 단위를 저장하는 컬럼이 따로 있고, 상세 주소를 저장하는 컬럼이 있는 게 더 효율적일까? 
아니면 주소 컬럼은 하나로 유지한 채로 사용자에게 보여줄 때 split을 해서 가장 앞의 정보를 보여주는 게 효율적일까? 

위의 제약사항과 내 생각(고민)을 고려해서 기본적인 틀(설정과 비즈니스 로직 등)을 잡아줘
```

### 4. 관리자 회원 관리 API
```
지금은 관리자가 사용할 회원 조회 API가 필요해 조건은 아래와 같아

1. 조회는 pagination 기반
2. 수정은 암호, 주소에 대해서만 가능. 한 가지씩 수정도 가능하고, 두 가지를 동시에 수정도 가능
3. 시스템 관리자 API 들의 인증 수단은 basic auth 기반
- 사용자명: admin
- 암호: 1212

내 생각엔 pagination은 QueryDsl을 사용해서 구현하는 게 좋을 것 같고, 수정은 값이 입력되었는지를 확인한 후에 수정하는 게 좋을 것 같아.
이 기능은 service의 로직보다는 authentication과 authorization에 대한 고민이 더 필요할 것 같아.

내가 생각하는 방향이 맞는지, 혹은 더 나은 방향이 있는지 설명해줘
```
